# 弃用 CPA 主线路径并切换到 Codex-First 决策

## 决策背景

当前仓库中的网关实现，已经大量偏向 `https://chatgpt.com/backend-api/codex`，但仍保留了多处 `CPA` 兼容思路：

- 历史兼容开关
- 部分“少状态、少重试、单跳上游”的兼容逻辑
- 文档与注释中“对齐 CPA”的历史表述

这套路径的优点是兼容面更宽，但它和官方 Codex 的真实会话心智并不一致。

当前主要问题：

- 同一会话与账号之间缺少单一事实源
- 自动切换账号时没有显式线程换代语义
- 部分兼容分支会牺牲会话连续性来换取可用性
- 产品目标已经从“先兼容能跑”转为“尽量对齐官方 Codex 行为”

## 决策结论

项目后续主路径改为 `Codex-First`，不再以 `CPA` 兼容行为作为主线路径语义。

具体结论如下：

1. `CPA` 相关开关不再作为长期主线路径保留。
2. `CPA` 相关开关不再作为官方排障建议。
3. `conversation_id` 代表本地会话，不再直接承担跨账号共享上游线程的职责。
4. 同一会话在任一时刻只绑定一个活动账号。
5. 手动切换账号会切线程，自动切换账号也必须切线程。
6. `ordered` / `balanced` 仅保留为首次绑定与重新绑定策略。

## 决策理由

### 为什么不再继续对齐 CPA

- `CPA` 的目标更偏兼容优先，而不是官方会话语义优先。
- 它更强调减少头部、减少状态、减少重试。
- 它不适合作为多账号会话绑定模型的长期基础。
- 继续围绕 `CPA` 打补丁，会让产品长期停留在“能跑但不像 Codex”的状态。

### 为什么改成 Codex-First

- 当前产品主要目标已经是稳定模拟官方 Codex 行为。
- 用户更关心会话一致性、账号切换语义和线程连续性。
- 只有先把语义统一，后续才能继续处理 Cloudflare 与 transport 指纹问题。

## 影响范围

该决策会影响以下部分：

- gateway 路由策略
- 会话与账号绑定模型
- `prompt_cache_key` / `session_id` / `turn-state` 的使用方式
- 设置页网关配置项
- 环境变量说明
- FAQ 与排障文档
- 兼容测试与回归用例

## 兼容与迁移原则

- 文档先行，代码随后跟进。
- 在代码正式完成迁移前，文档必须明确区分“当前行为”和“迁移目标”。
- 如果迁移中发现重大兼容问题，优先通过版本回滚处理，而不是重新把 `CPA` 提回长期主路径。
- 旧兼容开关移除后，不再重新引入同类设置页与环境变量入口。
