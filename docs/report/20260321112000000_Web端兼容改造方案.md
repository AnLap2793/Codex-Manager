# Web 端兼容改造方案

## 目标
- 保持桌面端现有稳定性，不为兼容 Web 破坏 Tauri 体验。
- 让 Web 端核心业务功能稳定可用，避免“入口能点，一执行就报错”。
- 明确桌面专属能力与共通业务能力的边界，减少后续回归。

## 结论
- 当前稳定的 Web 运行方式，不是任意静态托管，而是由 `codexmanager-web` 提供静态资源和同源 `/api/rpc` 代理的运行方式。
- 账号、平台密钥、日志、网关设置、Web 访问密码这类核心业务，前端已基本具备 Web 映射；真正不该强行兼容到 Web 的，是服务启停、托盘、应用更新、打开本地目录这类本地能力。
- 因此更好的方案不是“让所有桌面命令都能在 Web 执行”，而是“把能力边界写死”：
  - 共通业务统一走 service RPC。
  - 桌面专属能力只在 Tauri 暴露。
  - Web UI 依据运行时能力统一降级。

## 已确认事实

### 1. 前端是静态导出
- `apps/next.config.ts` 使用了 `output: 'export'`。
- 这意味着 `apps` 产物本身不是一个带 Next API 的 Web 服务。

### 2. 非 Tauri 环境固定走同源 `/api/rpc`
- `apps/src/lib/api/transport.ts` 中，非 Tauri 运行时会走 `invokeWebRpc`。
- `invokeWebRpc` 固定向 `"/api/rpc"` 发送 JSON-RPC 请求。
- 所以前端本身依赖一个外层 Web 容器来提供同源代理。

### 3. `/api/rpc` 由 `codexmanager-web` 提供
- `crates/web/src/main.rs` 挂载了 `/api/rpc` 和静态资源服务。
- `crates/web/src/service_gateway.rs` 会把请求转发给 `codexmanager-service`，并由服务端注入 `x-codexmanager-rpc-token`。
- 这也说明当前设计并不是让浏览器直接访问 service RPC。

### 4. Web 映射已覆盖大部分核心业务 RPC
- `apps/src/lib/api/transport.ts` 的 `WEB_COMMAND_MAP` 已覆盖：
  - 账号列表、删除、导入导出、登录、用量。
  - 平台密钥的增删改查、启停、模型列表、读取明文。
  - 请求日志查询、汇总、清理。
  - 网关设置、监听设置、启动快照、应用设置。
- 所以 Web 端“很多业务功能都不能用”，主因并不主要是业务 RPC 完全缺失。

### 5. 当前缺失的 Web 映射主要是桌面专属命令
- 当前前端调用里，未进入 `WEB_COMMAND_MAP` 的主要是：
  - `service_start`
  - `service_stop`
  - `app_close_to_tray_on_close_get`
  - `app_close_to_tray_on_close_set`
  - `app_update_check`
  - `app_update_prepare`
  - `app_update_launch_installer`
  - `app_update_apply_portable`
  - `app_update_status`
- 这些能力本身就更接近桌面运行壳，不应该作为 Web 必须支持的能力。

### 6. UI 已有部分降级，但规则分散
- `apps/src/components/layout/header.tsx` 只在桌面端展示服务启停控件。
- `apps/src/app/settings/page.tsx` 已对更新按钮做了 `!isDesktopRuntime` 禁用，并让托盘开关依赖 `closeToTraySupported`。
- 但这些判断分散在各页面里，不是统一能力模型，后续仍容易漏。

## 当前 Web 易出错的根因

### 根因一：运行方式契约是隐式的
- 非 Tauri 就会走 `/api/rpc`。
- 但 `apps` 本身不提供 `/api/rpc`。
- 所以只要以下运行方式没有补代理，就容易出现大量报错：
  - 直接打开 `apps/out`
  - 直接运行 `pnpm dev` / `pnpm start`
  - 把静态资源部署到普通 CDN / Nginx，但没有转发 `/api/rpc`

### 根因二：前端只区分“是不是 Tauri”，没有区分“是不是受支持的 Web 壳”
- 当前核心判断只有 `isTauriRuntime()`。
- 实际上至少应区分三种形态：
  - `desktop-tauri`
  - `web-gateway`
  - `unsupported-web`
- 现在把后两者都当作“普通 Web”处理，会让 unsupported 环境表现得像“功能坏了”。

### 根因三：能力判断散落在页面里
- 是否显示按钮、是否禁用按钮，分散在 `Header`、`Settings` 等页面和组件里。
- 一旦新增功能或弹窗，就容易再次出现“桌面正常、Web 某个角落一点击就炸”。

### 根因四：缺少桌面 / Web 双运行壳冒烟验证
- 现在能说明：
  - `apps` 静态构建可以通过。
  - `codexmanager-web` 自身测试可以通过。
- 但还缺少“桌面一轮 + Web 一轮”的能力冒烟清单，很多兼容问题只能靠人工撞出来。

## 不建议的方案

### 不建议：让浏览器直接访问 `codexmanager-service`
- 这会把 RPC 认证、CORS、服务暴露、局域网边界等问题都推到前端。
- 从当前 `codexmanager-web` 的实现看，项目已经选择了“Web 壳代理 service”这条路，继续沿着这条路完善更稳。

### 不建议：继续用页面级 if/else 零散补洞
- 短期能止血，但每新增一个入口都要重新记忆桌面/Web 差异。
- 维护成本会持续升高，而且很难防止后续回归。

## 推荐方案

### 1. 保留现有总体方向
- 桌面端：`Tauri + invoke`
- Web 端：`codexmanager-web + /api/rpc`

不要把“让纯浏览器直连 service”当成这次兼容改造的一部分。

### 2. 新增统一运行模式
建议前端引入：

```ts
type RuntimeMode =
  | "desktop-tauri"
  | "web-gateway"
  | "unsupported-web";
```

判定方式建议：
- 命中 Tauri：`desktop-tauri`
- 非 Tauri 时，先探测一个同源探针接口，例如 `/api/runtime` 或 `/__runtime`
- 探针成功：`web-gateway`
- 探针失败：`unsupported-web`

### 3. 新增统一能力模型
建议前端统一维护：

```ts
type RuntimeCapabilities = {
  mode: RuntimeMode;
  canManageService: boolean;
  canSelfUpdate: boolean;
  canCloseToTray: boolean;
  canOpenLocalDir: boolean;
  canUseBrowserFileImport: boolean;
  canUseBrowserDownloadExport: boolean;
  rpcBaseUrl: string;
};
```

页面和组件不再直接猜运行环境，只依赖能力集。

### 4. 把能力边界写死

#### 共通业务能力
这些能力应在桌面端和 Web 端都能正常：
- 账号列表、删除、批量导入、批量导出、登录授权、用量刷新。
- 平台密钥的增删改查、启停、读密钥、模型列表。
- 请求日志查询、汇总、清理。
- 网关设置、监听设置、环境变量覆盖、Web 密码。

#### 桌面专属能力
这些能力不需要强行做成 Web 可执行，只需要在 Web 上正确隐藏或禁用：
- 启动 / 停止本地 service
- 关闭到托盘
- 下载并替换桌面应用更新
- 打开本地目录、日志目录、文件管理器

### 5. 给 Web 壳补一个运行时探针
建议在 `crates/web/src/main.rs` 新增一个轻量接口，例如 `/api/runtime`，返回：
- 当前模式
- `rpcBaseUrl`
- 能力集

这样前端启动时就能第一时间知道：
- 当前是不是正确的 Web 运行壳
- 当前有哪些能力可用
- 当前报错到底是业务错误还是部署方式错误

### 6. 让 `rpcBaseUrl` 可配置
建议把 `apps/src/lib/api/transport.ts` 里的固定 `"/api/rpc"` 改成：
- 默认：`/api/rpc`
- 可选：由运行时探针或环境变量覆盖

这样更利于：
- 本地开发态代理
- 将来 Web UI 与 `codexmanager-web` 分离部署

### 7. unsupported-web 要有专门提示
`apps/src/components/layout/app-bootstrap.tsx` 在探测到 unsupported-web 时，不应继续给出泛化的“服务初始化失败”。
应该直接提示：
- 当前页面缺少 `/api/rpc` 代理
- 请通过 `codexmanager-web` 打开，或在反向代理中转发 `/api/rpc`

## 建议改动文件

| 文件 | 改动建议 |
| --- | --- |
| `apps/src/lib/api/transport.ts` | 增加 `RuntimeMode`、探针读取、可配置 `rpcBaseUrl`、统一能力读取 |
| `apps/src/types/index.ts` | 增加 `RuntimeCapabilities` 类型 |
| `apps/src/components/layout/app-bootstrap.tsx` | 启动时探测运行壳；对 unsupported-web 输出专门错误页 |
| `apps/src/components/layout/header.tsx` | 不再自行判断 Tauri，改为读取能力集 |
| `apps/src/app/settings/page.tsx` | 更新、托盘、本地目录等入口改为能力驱动 |
| `apps/src/app/accounts/page.tsx` | 导入导出入口根据能力决定表现和提示文案 |
| `crates/web/src/main.rs` | 新增 `/api/runtime`，统一暴露 Web 壳能力 |
| `docs/report/20260310122606850_运行与部署指南.md` | 明确“受支持的 Web 运行方式”和 `/api/rpc` 代理要求 |

## 分阶段实施建议

### P0：先止血
- 在启动阶段识别 unsupported-web，并显示明确错误页。
- 在文档中写清楚：Web UI 必须通过 `codexmanager-web` 或等价代理运行。
- 给 `transport.ts` 增加可配置 `rpcBaseUrl`。

### P1：做结构化改造
- 增加 `RuntimeMode` 和 `RuntimeCapabilities`。
- 所有页面改为从能力集读取支持状态。
- 收敛掉散落的 `isTauriRuntime()` 页面逻辑。

### P2：补验证
- 增加桌面端冒烟清单。
- 增加 `codexmanager-web` 运行壳冒烟清单。
- 至少覆盖账号、密钥、日志、设置、Web 密码，以及 Web 下桌面专属按钮不报错。

## 验收标准
- 桌面端现有功能不回退。
- Web 端通过 `codexmanager-web` 打开时，核心业务能力可正常使用。
- 不受支持的 Web 运行方式下，不再出现一堆业务按钮报错，而是明确提示运行方式错误。
- 桌面专属能力在 Web 端不会再误触发“当前环境不支持该操作”这类硬错误。

## 本次扫描的直接验证
- `pnpm run build`（目录：`apps`）通过，说明前端静态导出链路正常。
- `cargo test -p codexmanager-web` 通过，说明 Web 壳的核心路由和认证单测正常。

## 最终建议
- 如果目标是“发布一个稳定可用的 Web 版”，建议继续以 `codexmanager-web` 作为唯一受支持的 Web 运行壳，并把这个约束写进代码和文档。
- 如果目标是“让纯静态托管也能正常用”，那已经不是简单补前端判断，而是要新增正式的 Web BFF / 代理服务；在当前架构下，这应视为单独项目级改造。
